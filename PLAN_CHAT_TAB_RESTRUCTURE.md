## Context

The current Briefing tab (`Screen.Briefing` route, `BriefingScreen.kt`) is a passive
read-only view: it loads the latest AI report and suggestion cards from the database and
lets the user approve or reject them. The AI report is generated by `AiAnalysisWorker`
on a 24-hour WorkManager schedule and has no on-demand trigger from the UI.

The goal is to replace this tab with an interactive Chat tab that acts as a
**tool launcher**: a set of action cards at the top, and a scrollable conversation
thread below where tool outputs appear inline. Two tools ship in this milestone:

1. **Briefing tool** — triggers on-demand briefing generation (reuses existing
   `PatternAnalyzer` + `ReportGenerator` logic but bypasses WorkManager).
2. **Rule Creation tool** — user types a natural-language description, the system
   extracts structured fields via per-field LLM calls, surfaces a review form, accepts
   follow-up refinement messages, and on approval writes the rule to `RuleDao` with
   `source="ai"` and `status="pending_review"`.

Key architectural facts confirmed by code reading:

- `MainActivity.kt:246` — NavHost `composable(Screen.Briefing.route)` is a one-liner.
  Swapping the composable is a single-line change.
- `Screen.Briefing` is referenced in `MainTab.all` (label "Briefing", icon Home),
  in the nav `popUpTo` call in `LithiumBottomBar`, in `MainActivity.onCreate` (start
  route), and in `BriefingViewModel` (WorkManager flow watcher).
- `LlamaEngine` uses `CONTEXT_SIZE = 512` and `MAX_OUTPUT_TOKENS = 16` — tuned for
  classification. Generative field extraction needs a larger context window and more
  output tokens.
- `LlamaCpp.runInference` accepts arbitrary `maxTokens` already — no JNI changes needed.
- `ReportGenerator` is template-based (no LLM), so on-demand briefing is fast and
  requires only extracting the worker's steps 3-5 into a callable function.
- `RuleCondition` sealed class is already modular: `PackageMatch`, `ChannelMatch`,
  `CategoryMatch`, `NotFromContact`, `CompositeAnd`. Per-field extraction maps cleanly.
- `Rule` entity has `source` and `status` columns. `status="pending_review"` and
  `source="ai"` are valid values per the schema and existing `RuleDao` usage.
- The existing `AddRuleScreen` / `AddRuleViewModel` handle a simpler user-driven form.
  We do not reuse these directly, but the field layout and save logic are the model.
- `WorkScheduler.runNow()` already enqueues an on-demand `AiAnalysisWorker`. The
  Briefing tool does NOT use this — it calls the generation logic directly to avoid
  the full classification pass.

## Approach (Recommended): Rename Route, New UI Package, Extract BriefingService

The cleanest path renames `Screen.Briefing` to `Screen.Chat` in `MainActivity.kt`
only (no other nav changes), creates a new `ui/chat/` package, and extracts
the on-demand report generation logic into an injectable `BriefingService` that
both `AiAnalysisWorker` and the new `ChatViewModel` can call.

**Why this wins over alternatives:**

- Minimal nav surgery: one route rename, one composable swap.
- Briefing generation logic lives in one place (`BriefingService`) — no duplication
  between the worker and the chat tab.
- The per-field LLM extractor is a thin wrapper around the existing `LlamaEngine`
  with an expanded context window, keeping the JNI layer untouched.
- Rule form state is self-contained in `ChatViewModel` — no dependency on
  `AddRuleViewModel` machinery.

## Alternatives Considered

### Alternative A: Keep `Screen.Briefing` route, add Chat as a sixth tab

Add a new `Screen.Chat` route and tab, keep Briefing as-is.

Rejected because: the user explicitly wants to replace the Briefing tab, not add
alongside it. Six tabs is crowded on small screens. The existing briefing report view
can live in the chat thread as a tool output, making the Briefing tab redundant.

### Alternative B: Inline everything in `ChatViewModel`, no `BriefingService` extraction

Put briefing generation logic directly in `ChatViewModel` by depending on
`PatternAnalyzer` and `ReportGenerator` directly.

Rejected because: `AiAnalysisWorker` also calls these components. Extracting a
shared `BriefingService` prevents the logic from drifting out of sync and makes
the worker testable without WorkManager harness. The extraction cost is low.

## LLM Model Capability Assessment

**Current models (from `LlamaEngine` comments):** SmolLM-135M, Qwen2-0.5B,
Qwen2.5-1.5B — all INT4 quantised.

**Reliable tasks for these models:**
- Extracting a package name hint from "mute Slack DMs" -> `"com.slack"` (few-shot,
  constrained output, single value). These models handle this adequately at Q4.
- Extracting an action word ("mute", "suppress", "queue") -> mapped to enum.
- Extracting a boolean (contact filter yes/no) -> very reliable.

**Unreliable tasks:**
- Disambiguating channel IDs (model doesn't know Slack's internal channel names).
- Inferring time-window conditions (the `RuleCondition` model has no time field;
  skip for now and surface in form as a manual field).
- Multi-entity extraction in one pass from a single model call.

**Decision:** Per-field extraction with short focused prompts is within these models'
capability for the fields that map to `RuleCondition`. Fields the model can't resolve
reliably (channel IDs) are left blank in the draft form for user completion. Do NOT
route to Cairn API server as a fallback in this milestone -- the Cairn dependency adds
network coupling and auth complexity that is out of scope. Instead, low-confidence
fields are surfaced blank with an explanatory hint.

**Context window:** Field extraction prompts need ~300-400 tokens (system + few-shot +
input). The current `CONTEXT_SIZE = 512` is just sufficient. Bumping to 1024 in the
generative path gives headroom without requiring a model reload. Since `LlamaEngine`
is a singleton with a single model handle, the context size is set at `loadModel` time.

**Resolution:** Add a `generate()` method to `LlamaEngine` that reuses the existing
model handle but calls `runInference` with the caller-supplied token count. Track the
context size at load time. Since `AiAnalysisWorker` releases the model after each pass
(step 8), the model handle is 0L when the chat tab calls in -- a fresh load with the
right context size (1024) is the common case. Document this in `RuleExtractor`.

## Per-Field LLM Extractor Contract

### Fields extracted

| Field | RuleCondition type | Prompt output format | Fallback if blank |
|-------|-------------------|---------------------|-------------------|
| App match | PackageMatch.packageName | Single package name string | User selects from dropdown |
| Channel match | ChannelMatch.channelId | Single channel ID string | Left blank, user fills |
| Category | CategoryMatch.category | One of known labels | Left blank |
| Contact filter | NotFromContact (boolean) | `yes` or `no` | Default `no` |
| Action | Rule.action | `suppress` or `queue` | Default `suppress` |

### Prompt template shape (per field)

```
[SYSTEM] You are a rule extraction assistant. Given a user notification rule
description, extract only the requested field. Reply with the value only.
No explanation. If you cannot determine the value, reply: unknown

[USER]
Field: app package name
Description: "mute Slack DMs except from Sarah"
Known packages: com.slack, com.whatsapp, com.google.android.gm
Reply:

[ASSISTANT]
```

The "known packages" list comes from `NotificationRepository.getDistinctPackageNames()`,
same as `AddRuleViewModel`. This provides grounding so the model does not hallucinate.

### Output parsing

- Trim, lowercase, check against known values or known package list.
- If output equals `"unknown"` or is empty -> field is `null` in draft.
- For action: if output contains `"queue"` -> queue; otherwise suppress.
- For contact filter: if output contains `"yes"` or `"contact"` -> NotFromContact.

### Error/ambiguity handling

- `null` field in draft -> form field shows placeholder: "Could not determine -- fill in".
- Extraction runs sequentially (one field at a time, each inside `inferenceMutex`).
  The UI shows a progress indicator while extraction runs.
- If `LlamaEngine.isModelLoaded()` returns false after load attempt -> all fields are
  null, form shows banner: "On-device model not available -- fill in the fields manually."

### Follow-up refinement

Follow-up messages trigger a re-extraction pass using the same per-field prompts but
with the follow-up appended to the original description. The form is updated with any
newly non-null fields. Fields the user has manually edited are preserved (tracked in
`userEditedFields`) and not overwritten by re-extraction.

## Risks and Mitigations

### Risk 1: LlamaEngine context window too small for extraction prompts

The current `CONTEXT_SIZE = 512` may truncate field extraction prompts that include
a long known-packages list (dozens of package names).

**Mitigation:** In `LlamaEngine.loadModel`, store the context size. If `generate()`
is called and the stored context size is below 1024, log a warning. In practice, the
model is unloaded after each worker run, so the chat tab re-loads it with 1024. In
`RuleExtractor`, truncate the known-packages list to the 20 most-recently-seen
packages to keep prompt length bounded.

### Risk 2: Small models hallucinate package names not in the device's history

A 135M or 500M parameter model may output a plausible but wrong package name.

**Mitigation:** In `RuleExtractor.extract()`, after extracting the package name field,
validate the output against `NotificationRepository.getDistinctPackageNames()`. If not
in the list, treat as null. The form field is then blank and the user selects from a
dropdown populated with the same list -- the same pattern as `AddRuleViewModel`.

### Risk 3: `inferenceMutex` contention if worker runs during extraction

`AiAnalysisWorker` and `ChatViewModel` both use the same `LlamaEngine` singleton and
its `inferenceMutex`. If the 24h worker fires during a rule extraction session,
inference calls queue behind the mutex.

**Mitigation:** Tolerable. The mutex prevents corruption. The worker only runs under
charging/constraint conditions by default. Log the wait time so it is observable.

### Risk 4: BriefingService model loading during on-demand invocation

The worker loads models at start and releases at end. The chat tab must handle loading.

**Mitigation:** `ChatViewModel.invokeBriefing()` ensures `llamaEngine.loadModel()` is
called before `briefingService.generateReport()`. Note: `ReportGenerator` is
template-based and does not itself call `LlamaEngine`. Model load is only needed if
the briefing path were to use LLM-based summarisation in the future. For this
milestone, `invokeBriefing()` does NOT need to load the LLM -- it only needs
`PatternAnalyzer` + `ReportGenerator`. Model loading is only needed in
`startRuleCreation()`.

### Risk 5: Rule saved with status="pending_review" is invisible to RuleEngine

`RuleRepository.approvedRules` filters for `status="approved"` only. A pending_review
rule will not be evaluated by `RuleEngine` until the user promotes it.

**Mitigation:** This is the intended behaviour. Surface it clearly: show a system
message "Rule saved for review -- go to Rules tab to activate it." An optional
"Activate now" button in the chat thread that writes `status="approved"` directly
is a stretch goal for a follow-up.

### Risk 6: Leftover BriefingScreen.kt confuses future maintainers

**Mitigation:** Add a comment at the top of `BriefingScreen.kt` noting it is no longer
used in navigation but retained for the existing test suite. Schedule deletion as a
follow-up issue in the backlog.

### Risk 7: LlamaCpp native layer caps maxTokens below caller's value

The `generate()` method passes `maxTokens` to `LlamaCpp.runInference()`. The JNI
implementation at `app/src/main/cpp/llama_jni.cpp` was not read.

**Mitigation (validation required):** Before implementing `RuleExtractor`, verify that
`LlamaCpp.runInference(handle, prompt, maxTokens)` respects the token count argument.
Field extraction needs ~20-50 output tokens; if the native layer caps at 16, extend it.

**High confidence:** nav changes (trivial), `BriefingService` extraction (mechanical
refactor of worker steps 3-5), `LlamaEngine.generate()` addition (thin wrapper over
existing JNI call), `RuleRepository.insertRule` with no schema changes (confirmed).

**Medium confidence:** per-field LLM extraction quality on the 135M/500M models. The
package-name grounding approach (validate against known list) is a hard safety net
regardless of model quality. Extraction degrades gracefully by leaving fields blank
rather than crashing or inserting garbage rules.

**Lower confidence:** channel ID extraction. Channel IDs are internal Android strings
(`"email_notifications"`, `"dm_alerts"`) that the model has no way to know without
context. Recommend leaving the channel field blank by default and instructing users to
consult the Notification Channels setting for the app. Acceptable trade-off.
